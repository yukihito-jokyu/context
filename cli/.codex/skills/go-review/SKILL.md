# context/cli Go Review

このスキルは `context/cli` に対する日常の差分レビュー専用です。
主対象は Go の変更差分です。レビューは常に差分起点で行い、必要なときだけ周辺ファイルを追加で読みます。

`README.md` や `docs/*.md` は原則レビュー対象外です。
ただし、Go の変更によって説明が明確に古くなっている場合だけ、更新漏れとして指摘します。

## 1. レビューの目的

- `context deploy` を中心とした CLI 実装が、今後の拡張に耐える責務分離を保っているか確認する
- `context` リポジトリ固有の仕様を壊していないか確認する
- 一般的な Go の悪手や保守性低下を早期に検出する
- 指摘が必要な事項を、重大度順かつ根拠付きで利用者に返す

## 2. レビューの進め方

### Step 1. 差分の確認

まず差分を確認します。確認順は次の通りです。

1. `git diff --cached`
2. `git diff`
3. 必要なら比較対象ブランチとの差分

差分が取れたら、まず変更ファイル一覧と変更の主題を把握します。

### Step 2. レビュー対象の絞り込み

- 主対象は `*.go`
- Go 差分に直接関係する説明更新漏れが見える場合のみ `README.md` や `docs/*.md` も補助的に確認する
- 指摘の根拠が必要な場合だけ周辺ファイルや関連テストを読む

### Step 3. context/cli 固有観点の確認

まずこの章を優先して確認します。一般 Go の指摘より優先度は上です。

### Step 4. 一般 Go チェックリストの確認

context/cli 固有観点で重大な問題がないかを見た後、一般 Go の観点を確認します。

### Step 5. 結果の整理

- 問題がある場合は findings-first で報告する
- 重大度順に並べる
- 各指摘には `ファイル/行`、`何が問題か`、`このプロジェクト方針にどう反するか`、`修正方針` を書く
- 修正コード例は必要なときだけ付ける
- 指摘なしでも、未実行テスト、未確認分岐、差分外で見ていない範囲などの残留リスクを短く書く

## 3. context/cli 固有チェックリスト

### 3.1 `cmd` の責務

- `cmd` パッケージが入出力とフロー制御に寄っているか
- `cmd` が探索、解決、ファイルコピー、候補収集などの実処理を抱え込みすぎていないか
- 依存差し替えしやすい形になっているか
- テストのために必要な注入ポイントが壊れていないか

### 3.2 `internal/*` の責務分離

- `internal/filesystem`、`internal/project`、`internal/prompt`、`internal/session` などの責務が混ざっていないか
- 将来の `deploy` 拡張時に差し替えや追加がしやすい境界になっているか
- 早い段階で不要な abstraction を増やしていないか
- interface が「利用側の必要」ではなく「なんとなく将来使いそう」で導入されていないか

### 3.3 `context` ルート探索

- `CONTEXT_REPO`、実行ファイル相対、親探索の優先順を崩していないか
- エラー内容が利用者にとって十分具体的か
- 不正なルート指定時に、足りない要素や修正方法が分かるか
- 探索ロジックが OS 依存の罠を増やしていないか

### 3.4 project 解決と suggestion

- `projects/<repo-name>` 前提を壊していないか
- 存在しない project 指定時のメッセージが利用者に優しいか
- suggestion が入力ミス補正として妥当か
- ディレクトリ列挙、ソート、候補表示の振る舞いが安定しているか

### 3.5 対話 I/O

- 標準入力、標準出力、標準エラー出力の責務が分かれているか
- プロンプト文言が曖昧でないか
- 非対話環境や入力不備時の失敗が分かりやすいか
- 表示処理と意思決定ロジックが密結合になっていないか

### 3.6 テストの方針

- テストが内部実装ではなく観測可能な振る舞いを見ているか
- ルート探索順序、project 解決、suggestion、セッション集約などの利用者価値に近い契約を検証しているか
- 実装詳細に強く結びつく brittle なテストになっていないか
- 将来の `deploy` 実装追加時にテスト構造を流用しやすいか

### 3.7 ドキュメント更新漏れ

- Go 差分によって `README.md` や `docs/*.md` の説明が明確に古くなっていないか
- 実装の判断が今後の保守性に影響するのに、どこにも理由が残っていない状態になっていないか

## 4. 一般 Go チェックリスト

この章は独立した一般 Go チェックリストです。
ただし優先順位は `context/cli` 固有観点の後です。

### 4.1 エラーハンドリング

- エラーを握りつぶしていないか
- 文脈を足すべき箇所で適切にラップしているか
- `errors.Is` / `errors.As` を使うべき比較で直接比較していないか
- 利用者向けエラーと内部原因の区別ができているか

### 4.2 制御フロー

- 早期 return によりネストを浅く保っているか
- `if` で終了する分岐の後に不要な `else` を置いていないか
- 正常系が右に流れすぎて可読性を落としていないか

### 4.3 abstraction と interface

- interface は利用側の必要に基づいているか
- テスト容易性の名目で不要な interface を増やしていないか
- 小さな CLI に対して過剰な層分割や抽象化をしていないか

### 4.4 命名と API の分かりやすさ

- 型名、関数名、変数名が役割を説明できているか
- `ID` などの頭字語が Go の慣習に沿っているか
- レシーバ名が短く一貫しているか
- export の要否が妥当か

### 4.5 データ構造と実装の素直さ

- nil と empty の扱いが意図通りか
- 分かりきったサイズに対して不自然な再 allocation を起こしていないか
- 複雑な helper を増やすより、素直な実装の方が読みやすい箇所で過剰設計していないか

### 4.6 テストの書き方

- テーブル駆動テストが有効な箇所で素直に使っているか
- 失敗時のメッセージから `got` / `want` や前提条件が追えるか
- 過剰なモック化でテストが実装依存になっていないか
- テスト名がケースの差を説明しているか

## 5. 条件付きチェックリスト

差分に並行処理、重い処理、性能が絡む実装が含まれる場合だけ有効にします。

### 5.1 並行処理

- `go func`、channel、`sync.*`、`context.Context` が追加・変更されているか
- 共有状態へのアクセスに race の余地がないか
- 同期ポイントや終了条件が明確か

### 5.2 goroutine lifetime

- 起動した goroutine がいつ終了するか説明できるか
- 呼び出し元の終了後にぶら下がらないか
- channel 待ちや送信でリークしないか
- cancel、close、timeout の責務が曖昧でないか

### 5.3 性能

- まずアルゴリズム、ループ回数、I/O 回数の無駄を確認する
- 同じファイル走査や `os.Stat` などを不必要に繰り返していないか
- CLI の文脈で低レベル最適化より先に、処理回数と設計上の無駄を潰しているか

### 5.4 低レベルなメモリ最適化

- ホットパスでもない箇所に細かい最適化を持ち込み、可読性を落としていないか
- 大量データや明確なボトルネックがある場合だけ、capacity 指定や不要コピー削減を検討しているか

## 6. 出力形式

レビュー結果は次の方針で出力します。

```markdown
[findings]
1. [severity] [file:line]
   - 何が問題か
   - このプロジェクト方針にどう反するか
   - 修正方針

[open questions]
- 必要な場合のみ書く

[residual risks]
- 指摘なしの場合でも、未実行のテスト、未確認の分岐、差分外で見ていない範囲を書く
```

実際の回答では、次のルールを守ります。

- 指摘がある場合は findings-first
- 重大度順に並べる
- 要約は短く、後段に置く
- 指摘がない場合は、その旨を明示した上で残留リスクを書く
