# テスト駆動開発

## 哲学

**核心の原則**：テストは実装の詳細ではなく、パブリックインターフェースを通じて振る舞いを検証すべき。コードは完全に変わってもよい。テストは変わらないべき。

**良いテスト**はインテグレーションスタイル：パブリックAPIを通じて実際のコードパスを実行する。システムが_何を_するかを記述し、_どう_するかではない。良いテストは仕様のように読める — 「ユーザーは有効なカートでチェックアウトできる」はどの機能が存在するかを正確に伝える。これらのテストは内部構造を気にしないため、リファクタリング後も生き残る。

**悪いテスト**は実装に結合している。内部の共同作業者をモックし、プライベートメソッドをテストし、外部の手段で検証する（インターフェースを使用する代わりにデータベースに直接クエリするなど）。警告サイン：リファクタリングしても振る舞いが変わっていないのにテストが壊れる。内部関数の名前を変更してテストが失敗するなら、そのテストは振る舞いではなく実装をテストしていた。

例とモッキングのガイドラインは[tests.md](tests.md)と[mocking.md](mocking.md)を参照。

## アンチパターン：水平スライス

**最初にすべてのテストを書き、次にすべての実装を書いてはいけない。** これは「水平スライシング」— REDを「全テストを書く」、GREENを「全コードを書く」と扱う。

これが生成するのは**粗悪なテスト**：

- 一括で書かれたテストは_実際の_振る舞いではなく_想像上の_振る舞いをテストする
- ユーザー向けの振る舞いではなく、ものの_形_（データ構造、関数シグネチャ）をテストしてしまう
- テストが実際の変更に鈍感になる — 振る舞いが壊れても通り、振る舞いが正常でも失敗する
- 実装を理解する前にテスト構造にコミットしてしまい、ヘッドライトの先を見越してしまう

**正しいアプローチ**：トレーサー弾による垂直スライス。1テスト → 1実装 → 繰り返し。各テストは前のサイクルで学んだことに対応する。コードを書いたばかりなので、どの振る舞いが重要でどう検証するか正確に分かっている。

```
間違い（水平）：
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

正しい（垂直）：
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  RED→GREEN: test3→impl3
  ...
```

## ワークフロー

### 0. 情報取得

作業内容として以下のコマンドでIssueの内容を取得してください:

```
gh issue view <issue-number>
```

### 1. 計画

コードベースを探索する際、プロジェクトのドメイン用語集を使用して、テスト名とインターフェースの語彙がプロジェクトの言語に一致するようにし、触れる領域のADRを尊重する。

コードを書く前に：

- [ ] ユーザーと必要なインターフェースの変更を確認
- [ ] ユーザーとテストする振る舞いを確認（優先順位付け）
- [ ] [深いモジュール](deep-modules.md)の機会を特定（小さなインターフェース、深い実装）
- [ ] [テスト可能性](interface-design.md)のためにインターフェースを設計
- [ ] テストする振る舞いをリスト（実装手順ではない）
- [ ] ユーザーの承認を得る

尋ねる：「パブリックインターフェースはどうあるべきか？ どの振る舞いを最も重視してテストすべきか？」

**すべてをテストできない。** ユーザーと正確にどの振る舞いが最も重要かを確認。テストの労力をクリティカルパスと複雑なロジックに集中させ、ありとあらゆるエッジケースではなく。

### 2. トレーサー弾

システムについての1つのことを確認する1つのテストを書く：

```
RED:   最初の振る舞いのテストを書く → テストが失敗
GREEN: 通すための最小限のコードを書く → テストが成功
```

これがトレーサー弾 — エンドツーエンドでパスが機能することを証明。

### 3. 段階的ループ

残りの各振る舞いについて：

```
RED:   次のテストを書く → 失敗
GREEN: 通すための最小限のコード → 成功
```

ルール：

- 一度に1つのテスト
- 現在のテストを通すのに十分なコードのみ
- 将来のテストを先読みしない
- 観察可能な振る舞いにテストを集中

### 4. リファクタリング

すべてのテストが通った後、[リファクタリング候補](refactoring.md)を探す：

- [ ] 重複を抽出
- [ ] モジュールを深化（複雑さをシンプルなインターフェースの背後に移動）
- [ ] 自然な場所でSOLID原則を適用
- [ ] 新しいコードが既存のコードについて何を明らかにするか考慮
- [ ] 各リファクタリングステップ後にテストを実行

**REDの間にリファクタリングしない。** まずGREENにする。

## サイクルごとのチェックリスト

```
[ ] テストは実装ではなく振る舞いを記述している
[ ] テストはパブリックインターフェースのみを使用
[ ] テストは内部リファクタリング後も生き残る
[ ] コードはこのテストに対して最小限
[ ] 投機的な機能は追加されていない
```
