# PRD: Context Repository Deploy CLI

## 問題提起

AI Agent向けコンテキストを複数の開発リポジトリで再利用したいが、`AGENTS.md`、`CLAUDE.md`、Skill (`SKILL.md`) の正本と配布先が分散すると、更新漏れ・重複管理・配布ミスが起きやすい。利用者は、どのプロジェクト向けコンテキストをどの配布先リポジトリへ反映するかを、毎回安全かつ一貫した手順で実行したい。

## 解決策

Context Repository を正本として、`context deploy` CLI で対象Projectを選択し、必要なSkillとドキュメントだけを対話的に配布できるようにする。CLIはファイル配布に責務を限定し、正本の探索、Skill重複解決、上書き確認、`CLAUDE.md` 生成を一貫して扱うことで、単一正本運用を崩さずに各開発リポジトリへ安全に反映できる状態を実現する。

## ユーザーストーリー

1. Context Repository の管理者として、プロジェクトごとのコンテキストを一元管理したい、なぜなら各開発リポジトリに正本を分散させたくないから
2. 開発リポジトリの保守者として、配布対象のProjectを明示的に選びたい、なぜなら誤ったプロジェクト設定を反映したくないから
3. CLI利用者として、引数なしでProject候補を一覧から選びたい、なぜなら正確なプロジェクト名を覚えていなくても配布したいから
4. CLI利用者として、存在しないProject名を指定したときに類似候補を見たい、なぜなら入力ミスを素早く修正したいから
5. CLI利用者として、配布先を現在の作業リポジトリとして扱いたい、なぜならその場で対象リポジトリへ反映したいから
6. CLI利用者として、配布先がGitリポジトリでない場合に警告を受けたい、なぜなら意図しないディレクトリへの配布を避けたいから
7. CLI利用者として、共通Skillとプロジェクト固有Skillをまとめて候補表示してほしい、なぜなら必要なSkillを一度の操作で選びたいから
8. プロジェクト管理者として、同名Skillがある場合はプロジェクト固有Skillを優先したい、なぜならプロジェクト固有の文脈を共通Skillより優先したいから
9. CLI利用者として、配布するSkillを毎回選択したい、なぜならリポジトリごとに必要なSkillだけを反映したいから
10. CLI利用者として、`AGENTS.md` を配布するか選びたい、なぜなら配布先の状況に応じて最低限の反映に留めたいから
11. CLI利用者として、`CLAUDE.md` を生成するか選びたい、なぜなら利用エージェントに応じて必要なファイルだけを用意したいから
12. CLI利用者として、既存Skillや既存ドキュメントがある場合に上書き確認を受けたい、なぜなら手動変更や既存運用を誤って消したくないから
13. CLI利用者として、選択したSkillを `.claude/skills/` と `.codex/skills/` の両方へ同じ内容で配布したい、なぜなら複数のAgent実行環境で同じコンテキストを使いたいから
14. Context Repository の保守者として、`SKILL.md` を唯一の正本として扱いたい、なぜなら要約や互換配置との内容差分を最小化したいから
15. AI Agent運用者として、READMEを配布対象から外したい、なぜなら配布物は実行コンテキストに必要な最小限に絞りたいから
16. CLI利用者として、CLIがLLM APIを呼ばずに完結してほしい、なぜなら配布処理は単純で予測可能なローカルファイル操作であってほしいから
17. 保守担当者として、Skill選択状態や履歴を保存しないでほしい、なぜなら状態管理を増やさず毎回明示的に確認したいから
18. 開発者として、Project探索・Skill解決・配布処理・対話UI・ファイル操作を分離したい、なぜならテストしやすく変更に強いCLIにしたいから

## 実装の決定

- CLIのMVPは `context deploy` と `context deploy <repo-name>` の2コマンドに限定する
- Context Repository ルートは環境変数、実行ファイル相対、カレントディレクトリからの親探索の順で解決する
- Projectは `projects/<repo-name>/` を単位として扱い、`README.md` は配布対象に含めない
- Skillは `SKILL.md` を必須要件とする単位ディレクトリとして扱い、メタデータファイルは導入しない
- Skill候補は共通Skillとプロジェクト固有Skillの双方から収集し、同名競合時はプロジェクト固有Skillを採用する
- 配布先はコマンド実行時のカレントディレクトリとし、可能であればGitリポジトリかを検査して警告を出す
- 選択されたSkillは `.claude/skills/` と `.codex/skills/` の両方へ同内容でコピーする
- `AGENTS.md` はProjectの正本から配布し、`CLAUDE.md` は初期実装では `AGENTS.md` と同一内容で生成する
- 既存Skillは `.claude/skills/<skill-name>` または `.codex/skills/<skill-name>` のいずれかが存在すれば既存扱いとし、上書き許可時のみ削除して再配置する
- 上書き確認はSkill、`AGENTS.md`、`CLAUDE.md` ごとに利用者へ明示的に求める
- 対話UIは実装量の少ないTUIライブラリを前提とし、初期実装ではチェックボックスと確認プロンプトを簡潔に組み立てられる構成を採用する
- アーキテクチャは、Project探索、Skill探索と重複解決、配布処理、対話UI、ファイル操作を独立モジュールとして分割する
- 配布CLIはLLM APIやREADME自動生成、配布履歴保存、選択状態保存を責務に含めない
- `projects/context` も他のProjectと同様に通常のProjectとして扱う

## テストの決定

- テストは内部実装ではなく外部から観測できる振る舞いを検証する。例として、Project解決結果、Skill優先順位、配布結果、上書き時の分岐を観測対象にする
- Project探索モジュールでは、存在するProject取得、存在しないProject名に対する類似候補提示、ルート探索順序の適用をテストする
- Skill解決モジュールでは、`SKILL.md` を持つディレクトリだけを候補化すること、共通Skillとプロジェクト固有Skillの競合時に後者が優先されることをテストする
- 配布モジュールでは、選択されたSkillが両配布先へ配置されること、配布先ディレクトリがなければ作成されること、上書き拒否時は既存内容を保持することをテストする
- `AGENTS.md` 配布と `CLAUDE.md` 生成では、生成有無の分岐、既存ファイルに対する確認、内容コピーの正しさをテストする
- ファイル操作モジュールでは、コピー、削除、存在確認を一時ディレクトリ上で検証し、副作用を局所化する
- 対話UIはUI部品の詳細よりも、選択結果と確認結果が上位フローへ正しく伝達される契約をテスト対象とする
- 類似テストの先行事例は現時点のコードベースには存在しない前提で、各モジュールの振る舞い中心テストを新設する

## スコープ外

- READMEの自動生成・更新
- SKILL.md の自動生成
- LLM API呼び出し
- `metadata.yaml` など追加メタデータ形式の導入
- 配布履歴の保存
- Skill選択状態の保存
- `projects/` 配下のカテゴリ階層化
- `deploy --dry-run` や `--only` など将来拡張コマンドの実装

## 補足事項

- 本PRDは `docs/requirements.md` と `docs/detailed-design.md` を統合して作成した
- Issueトラッカーへ公開するための運用情報やラベル語彙は、このリポジトリ内では確認できなかったため、本稿はまずリポジトリ内PRD文書として作成する
- 本機能の中心価値は「単一正本を保ったまま安全に配布すること」であり、配布CLI自体はコンテキスト生成ではなくファイル管理に徹する
