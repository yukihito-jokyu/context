# イシューへの変換

垂直スライス（トレーサー弾）を使って、計画を独立して着手可能なイシューに分割する。

イシュートラッカーとトリアージラベル語彙が提供されているべき — されていない場合は`/setup-matt-pocock-skills`を実行。

## プロセス

### 1. コンテキストの収集

会話コンテキストに既に存在するものから作業する。ユーザーが引数としてイシューの参照（イシュー番号、URL、またはパス）を渡した場合、イシュートラッカーから取得し、本文とコメントをすべて読む。

### 2. コードベースの探索（オプション）

まだコードベースを探索していない場合、現在のコードの状態を理解するために探索。イシューのタイトルと説明はプロジェクトのドメイン用語集の語彙を使用し、触れる領域のADRを尊重する。

### 3. 垂直スライスのドラフト

計画を**トレーサー弾**イシューに分割。各イシューは1つのレイヤーの水平スライスではなく、すべての統合レイヤーをエンドツーエンドで貫く薄い垂直スライス。

スライスは「HITL」または「AFK」のいずれか。HITLスライスは人間のインタラクションが必要（アーキテクチャの決定やデザインレビューなど）。AFKスライスは人間のインタラクションなしで実装・マージ可能。可能な限りAFKをHITLより優先。

<vertical-slice-rules>
- 各スライスはすべてのレイヤー（スキーマ、API、UI、テスト）を貫く、狭いが完全なパスを提供
- 完了したスライスは単体でデモまたは検証可能
- 少数の厚いスライスより多数の薄いスライスを優先
</vertical-slice-rules>

### 4. ユーザーに確認

提案された分割を番号付きリストで提示。各スライスについて：

- **タイトル**：短い説明的な名前
- **タイプ**：HITL / AFK
- **ブロック**：完了が先に必要な他のスライス（もしあれば）
- **対象ユーザーストーリー**：このスライスが対応するユーザーストーリー（元資料にある場合）

ユーザーに尋ねる：

- 粒度は適切か？（粗すぎ / 細かすぎ）
- 依存関係は正しいか？
- スライスを統合またはさらに分割すべきか？
- 正しいスライスがHITLとAFKにマークされているか？

ユーザーが分割を承認するまで反復。

### 5. イシュートラッカーにイシューを公開

承認された各スライスについて、イシュートラッカーに新しいイシューを公開。以下のイシューテンプレートを使用。これらのイシューはAFKエージェントが対応可能とみなされるため、指示がない限り正しいトリアージラベルで公開。

「ブロック」フィールドで実際のイシュー識別子を参照できるよう、依存順（ブロッカーを先に）で公開。

## 何を構築するか

この垂直スライスの簡潔な説明。レイヤーごとの実装ではなく、エンドツーエンドの振る舞いを記述。

具体的なファイルパスやコードスニペットは避ける — すぐに古くなる。例外：プロトタイプが文章よりも正確に決定をエンコードするスニペット（状態マシン、リデューサー、スキーマ、型の形）を生成した場合、ここにインラインし、プロトタイプから来たことを簡潔に記注。決定の豊富な部分のみを残す — 動作するデモではなく、重要な部分のみ。
